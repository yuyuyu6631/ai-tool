## 2024-06-25 - 为高频字符串处理函数添加 LRU 缓存
**经验：** 在处理工具目录分类、标签和搜索过滤等需要大量遍历的操作时，像 `_slugify` 这种涉及 Unicode 规范化和正则表达式的纯函数会成为显著的性能瓶颈。由于工具标签和类别的种类相对固定且重复度极高，无缓存的循环调用导致不必要的计算开销放大（时间复杂度为 O(N*M) 甚至更高）。
**行动：** 在存在重复调用的核心过滤和数据转换逻辑中，尤其是面对标签、分类这种离散程度低的数据时，应当首先考虑为纯转换函数（如 `_slugify` 等）增加 `@functools.lru_cache`，这可以以极小的内存代价大幅提升循环处理速度。
## 2024-05-18 - [缓存文本清洗函数并消除 N+1 查询匹配]
**经验：** 在 `catalog_service.py` 中，发现 `_repair_text` 和 `_slugify` 文本清洗函数会在列表循环中被针对相同的 tag 和 category 高频反复调用，导致显著的性能瓶颈。此外，用于搜索过滤的 `_matches_query` 接收原始查询字符串，在 N 个候选项的遍历中导致对同一搜索词产生冗余的 O(N*M) Token 分词预计算。
**行动：** 对于高频被重复访问的纯文本转换函数，使用 `@functools.lru_cache(maxsize=4096)` 进行缓存（测试显示处理 5 万次文本的时间由 0.26s 降至 0.009s）；将 `_matches_query` 重构为直接接受预计算好的 Token Groups（将搜索分词复杂度降为 O(N+M)）。这类对大数据量列表的高频清洗操作需优先排查 N+1 和冗余解析。
