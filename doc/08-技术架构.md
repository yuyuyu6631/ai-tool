# 08 技术架构

## 1. 架构结论

当前仓库已彻底抛弃早期的前端单体本地 Mock 方案（Vite Demo），全面升级为符合“核心验收准则”以及 MVO 愿景要求的全栈架构：

| 层级 | 当前实际采用 |
| --- | --- |
| 项目结构 | NPM Workspace (Monorepo) |
| 前端框架 | Next.js App Router (React 19) |
| 后端框架 | FastAPI (Python 3.10+) |
| 数据库层 | MySQL (通过 SQLAlchemy 2 + Alembic 驱动) |
| 样式中心 | Tailwind CSS 4 + UI 变量库 |
| 接口通信 | Next.js Server / Client Component 直接同源/反代 fetch 后端 API |
| 共享依赖 | `packages/contracts`（定义并分发前后端接口 Schema 类型约束） |
| 本地启动 | 提供 Docker Compose 一键启动 `(infra/docker)` 或本地双开 |

## 2. 为什么要更新这份文档

早期的文档依然记录着以 Vite 为基础进行 `src/app/utils/mockApi.ts` 拦截的环境，非常容易对现有开发者造成极度困扰和返工。经过系统化整理后，该文档目前已校准至最新的 Monorepo 真实架构基线。这套架构能更好支撑后续的打榜、计费以及真实抓取入库等企业级能力。

## 3. 架构分层详解

### 3.1 前端应用层 (apps/web)

构建模式基于 Server 侧直接渲染优先：
- 首页、分类页、工具详情页的大部分区域：属于 SSR/SSG，提升搜索引擎检索率（SEO），并能够直连缓存层拿取页面级数据。
- 搜索与多轮交互组件：属于 Client Components，利用 React Query 等工具做局部 CSR 数据重载，调用真实的 `/api/recommend` 后端服务。

### 3.2 后端应用层 (apps/api)

全面拥抱 Python 全栈生态，与前台实现彻底分离。
- `app/api/router.py` 暴露真正的 RESTful API 给前端消费。
- `models/models.py` 和迁移脚本保障所有内容表（`tools`, `scenarios`）在 MySQL 里的 ACID 稳定性。
- 服务层提供多段式过滤推荐功能：先找库里工具，再依靠大模型对入库结果做自然语言重排，最后送出。

## 4. 后续演进路线约束

- **不回头用假数据**：绝不随意再写 `.json` 塞在页面结构里；一切内容变更应交由 DB/API 端处理。
- **Redis 缓存集成**：已完整实现 `/api/recommend` 推荐接口结果缓存，利用 Redis 存储热门查询结果，配置可调整 TTL 进行请求降压；Redis 不可用时自动降级至实时计算，保证服务可用性。
- **类型一致性约束**：`packages/contracts` 已完整定义前后端共享类型契约，当前后端已使用，前端仍有类型重复定义；后续计划迁移前端从共享包直接导入类型，确保接口 Schema 一致性，减少重复维护工作。
- **环境隔离**：严格要求使用 `.env` 及对应镜像启动配置区分开发/生产逻辑（当前模板已预置）。
## 5. 当前前台展示策略补充

- 前台当前采用“尽量全量展示数据”的口径，`draft / published / archived` 都允许在公网页面展示。
- `status` 在当前阶段是内容状态标签，不作为前台访问门槛；不存在的 `slug` 才返回 404。
- 首页、工具目录、工具详情、榜单、场景都直接读取真实 API/数据库，不回退到 mock 或本地 JSON。
