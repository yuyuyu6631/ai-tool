# 13 前后端实现思路与任务拆分

## 1. 整理后的当前架构情况
通过剔除多余的旧演示版前端（现已归档到 `archive/drawer/legacy-frontend`）与不一致的跨层级文档目录（`docs/`），现在项目确立了最标准的 Monorepo 体系，目录结构如下：
- `apps/web/`: 唯一标准前端（基于 Next.js App Router + React Query）。
- `apps/api/`: 唯一标准后端（基于 FastAPI + Pydantic + SQLAlchemy + Alembic）。
- `packages/contracts/`: 前后端核心共享类型以及共享的契约结构。

## 2. 前后端实现设计思路
为了尽快跑通“工具录入 -> 推荐分发”的最小可运营闭环（MVO），我们的实现思路如下：

### 2.1 前端 (apps/web) 职责边界
- **Server Component 优先**：凡是涉及 SEO 的页面（如首页、榜单页、详情页、场景页），首屏数据均通过 SSR（或 SSG）直出，确保能够被搜索引擎快速抓取。
- **客户端数据获取**：涉及用户主动交互、翻页或高实时性请求的状态，通过 React Query 和 Fetch API，异步向 `apps/api` 请求。
- **去 mock 化**：必须彻底告别本地写死的 JSON 文件或 Mock 数据，所有工具、分类与场景数据，由 API 动态供给。

### 2.2 后端 (apps/api) 职责边界
- **真实数据驱动**：对接 MySQL 并使用 SQLAlchemy 进行真实义务数据的增删改查。榜单、工具详情、分类数据均应真实入库。引入 Redis 作为缓存层应对并发查询（特别是 `/api/recommend` 及首页请求）。
- **AI 与推荐层集成**：实现三段式推荐逻辑。首先依靠意图识别与规则从数据库找出强相关候选数据集；然后接入大模型通过 prompt 处理重排，最后拼接成结果输出。

## 3. 拆分出来的执行任务清单 (Task)

以下为接下来指导开发的具体任务拆分（开发时，可以按照本 Checklist 建立分支执行）。

### Task 1: 数据库真实数据接线（后端重点）
- [ ] **1.1 改写真实查询接口**：全面移除 `apps/api` 下服务层里面的静态返回逻辑，置换为通过真实会话链接到 MySQL 进行查询并返回对应的 Schema。
- [ ] **1.2 注入初始化真实数据（Seed）**：通过当前提供的 `seed.py` 脚本或通过管理员初始化接口，录入 10~20 条结构完整的测试工具记录、类目记录，以备接下来的调试和验证使用。

### Task 2: 前端抛弃假数据全面对接 API（前端重点）
- [ ] **2.1 环境信息与代理打通**：在 `apps/web` 下对接 `.env.local` 环境变量，配置好针对 FastAPI 代理请求的 Base URL（区分客户端调用与服务端调用）。
- [ ] **2.2 UI 层数据对接渲染**：修改首页、工具详情页、榜单页的抓取钩子，并使用之前跑通的真实 `apps/api` 接口。处理异常拦截和空白加载动画。
- [ ] **2.3 视觉素材排查转移**：确认归档目录 `archive/drawer/legacy-frontend` 中是否有品牌图片、Logo 未迁移的，若发现遗漏则迅速补充到 `apps/web/public/` 中。

### Task 3: 后台爬虫信息抓取与审核发布流程（服务层补充）
- [ ] **3.1 跑通定时/异步抓取入库逻辑**：实现 Python 后端触发 `crawl_jobs` 的机制，并让结果成功写入 `crawl_snapshots` 与 `tool_updates` 记录。
- [ ] **3.2 数据人工审核上架**：后端提供简单的工具状态流转接口（草稿 -> 已发布）。实现草稿数据向正式 Tool 数据的投递流转。

### Task 4: 容器化与本地快速联调（CI/CD 与基础架构）
- [ ] **4.1 Docker Compose 联调通过跑通**：优化并验证由于环境原因暂时没有测试的 Docker 链路。确保拉起指令 `docker-compose -f infra/docker/docker-compose.yml up --build` 能成功使得 web 端、API 端及 MySQL 等互相访问无碍。
- [ ] **4.2 完善多环境部署配置**：向后续的成员提供开发环境、测试环境及生产环境的不同配置文档说明。
