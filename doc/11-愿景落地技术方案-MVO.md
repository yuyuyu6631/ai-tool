# 愿景落地技术方案（MVO版）

## 1. 文档目标

本文用于把 [《愿景.md》](../goal/愿景.md) 从方向性描述转成可执行技术方案，回答三个问题：

1. 当前 Demo 能复用什么，不能承接什么
2. 面向最终目标，首个可落地的 MVO 应该包含哪些能力
3. 在技术路线不锁死的前提下，如何设计一条能持续演进的实现路径

本文不追求一次性确定所有技术细节，而是优先确定稳定的业务能力边界、数据边界和系统边界。

## 2. 核心判断

### 2.1 当前 Demo 的价值

现有代码和文档已经具备以下可复用资产：

- 前台信息架构已经成型：首页、榜单页、工具详情页、场景页、AI 搜索入口都已经跑通
- 已有一套基础内容模型：工具、场景、标签、推荐理由、编辑点评
- 已有可运行 UI 原型，适合直接复用视觉结构与交互骨架
- 已有基础文档体系，可以作为产品需求和页面设计的起点

### 2.2 当前 Demo 的局限

现状仍然是一个“展示型前台 Demo”，还不能支撑《愿景.md》里的“内容-流量-商业”闭环，核心缺口有：

- 数据层缺失：目前仍是本地 JSON 和 mock 数据，没有正式的数据源、内容库、版本管理、审核机制
- 运营层缺失：没有工具入库、内容生成、编辑审核、发布、下架、福利配置等运营后台能力
- 商业层缺失：没有合作工具配置、福利领取、线索沉淀、转化追踪
- 流量层缺失：没有 SEO 结构化输出、内容专题生产机制、埋点分析、投放归因
- 技术栈存在偏差：文档写的是 Next.js，但当前实现实际是 Vite + React Router，适合 Demo，不适合作为正式产品默认架构直接扩展

### 2.3 结论

不建议继续把当前 Demo 直接往“正式平台”上叠功能。更合理的做法是：

- 保留现有 Demo 的页面结构、组件表现、内容模板
- 先确认一版真正承接正式系统的主框架，再按模块逐步迁移
- 以“可运营的模块化单体”重建数据层、服务层和后台层
- 用 MVO 先验证“工具入库 -> 内容发布 -> 用户搜索/点击 -> 福利转化 -> 数据回流”的最小闭环

## 3. 产品目标重述

结合愿景文档，平台的最终目标不是做一个静态的 AI 工具导航站，而是做一个可信的 AI 智能体发现与转化平台。它至少要同时具备四类能力：

- 工具发现：让用户快速找到合适的 AI 工具/智能体
- 内容评测：让用户相信推荐结果，形成平台公信力
- 场景决策：让用户从“知道工具”走向“知道怎么选”
- 商业转化：让平台可以承接合作、福利、线索和后续服务

因此，首版系统设计不能只围绕“页面展示”展开，而要围绕“最小可运营闭环”展开。

## 4. MVO 定义

本文将首版目标定义为：

**MVO = Minimum Viable Operation，最小可运营版本。**

它和单纯的 MVP 不同。MVP 只要求“产品能看、能演示、能交互”；MVO 要求“业务可以开始真实运行，并能产生第一轮数据和反馈”。

### 4.1 MVO 必须验证的 5 个闭环

1. 工具闭环：工具可被采集、导入、审核、发布、更新
2. 内容闭环：围绕工具能持续产出详情、榜单、场景、专题内容
3. 推荐闭环：用户输入需求后，平台能返回站内工具，并引导进入详情或福利页
4. 商业闭环：合作工具可配置福利，用户可领取，平台可记录转化
5. 数据闭环：用户行为、内容表现、转化结果能回流，驱动后续榜单和选题调整

### 4.2 MVO 范围

MVO 内必须做：

- 公网前台站点
- 工具内容中心
- 场景/榜单/专题内容发布能力
- AI 搜索与推荐能力
- 简化版运营后台
- 福利/合作配置与领取链路
- 埋点分析与基础看板

MVO 内不做：

- 用户社区与公开评论体系
- 复杂账号体系和付费体系
- 全自动爬虫和全自动评测
- 多租户 SaaS 后台
- App、小程序、多端同步

## 5. 推荐的总体架构思路

### 5.1 架构原则

1. 先做“可运营单体”，不做微服务
2. 先做“结构化内容中心”，不做纯静态页
3. 先做“规则召回 + AI 重排”，不做重型推荐系统
4. 先做“人工审核的半自动流水线”，不做全自动采集
5. 先做“SEO 友好 + 内容可扩展”的前台架构

### 5.2 推荐架构形态

```text
用户浏览器
  |
  v
前台 Web（SSR/SSG）
  |
  +---- 页面渲染层：首页 / 榜单 / 工具详情 / 场景 / 专题 / 福利页
  |
  +---- BFF / API 层
          |
          +---- 工具目录服务
          +---- 内容发布服务
          +---- 搜索推荐服务
          +---- 商业合作服务
          +---- 埋点分析服务
          |
          +---- AI 模型网关
          |
          +---- 后台管理端
                  |
                  +---- 工具管理
                  +---- 内容管理
                  +---- 福利配置
                  +---- 审核发布
                  +---- 数据看板
          |
          +---- 数据层
                  +---- PostgreSQL / MySQL
                  +---- 对象存储
                  +---- 搜索索引（可选）
                  +---- 事件日志库
```

### 5.3 为什么是“模块化单体”

MVO 阶段的核心问题不是系统吞吐，而是业务闭环和团队执行效率。模块化单体的优势是：

- 一个代码库就能覆盖前台、后台、接口和基础任务
- 降低部署、排障、联调和权限复杂度
- 有利于小团队快速迭代
- 后续如果内容服务、推荐服务、采集服务变重，再拆分也不晚

## 6. 技术路线设计：定义能力，不锁死厂商

为了避免技术路线过早锁死，建议按“能力层”定义方案，而不是按“唯一技术”定义方案。

| 能力层 | 首选推荐 | 可替代方案 | 当前阶段建议 |
|------|----------|------------|--------------|
| 前台 Web | Next.js App Router | Nuxt / SvelteKit | 建议迁移到支持 SSR/SEO 的全栈框架 |
| BFF/API | Next.js Route Handlers / Server Actions | NestJS / FastAPI | MVO 阶段与前台同仓即可 |
| 主数据库 | PostgreSQL | MySQL | 优先 PostgreSQL，利于全文检索与结构化扩展 |
| 内容管理 | 自建轻后台 | Payload / Directus / Strapi | 优先自建简化后台，保证数据模型贴合业务 |
| 搜索 | PostgreSQL FTS + trigram | Meilisearch / Algolia / OpenSearch | MVO 先不用独立搜索引擎 |
| 对象存储 | S3 / R2 / OSS | 本地存储 | 图片、封面、导入文件统一走对象存储 |
| AI 网关 | 统一 Provider Adapter | 直接调用单一模型 | 必须抽象，避免绑死 DeepSeek/Claude |
| 任务调度 | Cron + 队列任务 | Trigger.dev / BullMQ | 首版只做导入、生成、重建索引等后台任务 |
| 埋点分析 | PostHog / Umami + Sentry | GA4 + 自建日志 | MVO 必须有行为分析和错误监控 |
| 身份认证 | 仅后台登录 | Clerk / Auth.js / Supabase Auth | 公网前台可匿名，后台必须有权限边界 |

### 6.1 正式框架定版建议

虽然路线不应该被厂商完全锁死，但当前阶段必须先确认一版“主框架”，否则会一直停留在 Demo 迭代状态。

建议正式主框架定版为：

- **前台与 BFF 主框架**：`Next.js App Router`
- **语言**：`TypeScript`
- **样式与组件**：`Tailwind CSS` + 现有可复用 React UI 组件
- **数据库**：`PostgreSQL`
- **ORM**：`Prisma`
- **后台认证**：`Auth.js` 或 `Clerk`
- **对象存储**：`S3 / R2 / OSS`
- **埋点与监控**：`PostHog` + `Sentry`

这套框架适合当前项目的原因是：

1. 与现有 Demo 同属 React 体系，前端组件迁移成本最低
2. 兼顾 SSR、SSG、SEO、动态路由和服务端接口能力
3. 一个工程内就能承载前台页面、后台页面和 API
4. 对内容平台、工具详情页、专题页、落地页这类 SEO 页面支持最好
5. 后续需要拆搜索、任务、推荐服务时，也不会推翻前台架构

这里的“先定框架”不是把所有技术一次锁死，而是先锁定主干：

- Web 层统一进 `Next.js`
- 数据主源统一进 `PostgreSQL`
- 前后台统一走同一套领域模型

搜索引擎、CMS 形态、模型供应商、队列系统都可以在这个主干上逐步替换。

### 6.2 为什么不继续以 Vite Demo 为主干

`Vite + React Router` 很适合快速出原型，但它不适合成为当前项目的正式主框架，原因是：

- SEO、metadata、静态生成、站点级路由能力需要额外补很多东西
- 后台、前台、接口、内容发布如果分散实现，后期结构会越来越散
- 你当前要做的不是单页交互产品，而是内容驱动、搜索驱动、转化驱动的平台

所以更合理的方式不是“继续在 Vite 上补齐所有能力”，而是“把 Vite Demo 作为 UI 资产，Next.js 作为正式系统主干”。

### 6.3 对现有 Demo 的处理建议

现有 `Next.js AI Tool Demo` 已归档到 `archive/drawer/legacy-frontend`，建议继续作为“前台视觉和交互资产来源”，而不是继续作为主工程演进。迁移策略如下：

1. 保留页面结构、组件层次、设计语言
2. 把静态 JSON 拆为正式领域模型
3. 把 mock API 替换成真实 BFF / API 层
4. 把 React Router 页面迁移到 SSR/SSG 路由体系
5. 把“内容写死”升级为“内容发布驱动”

## 7. 领域模型设计

MVO 必须先定义统一领域模型，否则后面内容、搜索、商业都接不上。

### 7.1 核心实体

| 实体 | 作用 | 说明 |
|------|------|------|
| Tool | 工具主档 | 平台的核心对象，所有详情、榜单、福利都围绕它展开 |
| ToolSnapshot | 工具采集快照 | 存原始采集/导入数据，用于溯源与对比 |
| Category | 分类 | 通用助手、写作办公、智能体平台等 |
| Tag | 标签 | 用于搜索、推荐和专题组合 |
| Scenario | 场景 | 如 PPT、写作、企业知识库助手 |
| Review | 评测内容 | 单工具评测，可人工编辑也可 AI 起草 |
| RankingList | 榜单 | 支持人工策展与规则计算混合生成 |
| TopicArticle | 专题文章 | 如“2026 最值得试的 AI PPT 工具” |
| BenefitOffer | 合作福利 | 优惠码、专属链接、免费额度等 |
| Lead | 用户线索 | 领取福利、预约演示、留下联系方式 |
| RecommendationLog | 推荐记录 | 记录用户 query、召回结果、点击结果 |
| PublishRecord | 发布记录 | 记录发布、下架、更新时间和责任人 |

### 7.2 关键关系

```text

Tool n---n Scenario

Tool 1---n BenefitOffer
Tool 1---n ToolSnapshot
Scenario 1---n TopicArticle
RankingList 1---n TopicArticle
BenefitOffer 1---n Lead
RecommendationLog n---n Tool
```

## 8. 核心模块方案

### 8.1 工具目录中心

目标：把“工具静态展示”升级为“可持续维护的工具知识库”。

最小能力：

- 支持手动创建工具
- 支持 Excel / CSV 导入工具
- 支持工具信息审核与发布
- 支持工具状态管理
- 支持工具版本更新和来源记录
- 支持标签、分类、场景绑定

首版建议流程：

1. 运营从 Excel 或表格导入候选工具
2. 系统生成草稿记录
3. AI 根据模板生成摘要、优缺点、适用场景草稿
4. 编辑人工审核后发布
5. 发布后自动生成详情页、榜单候选、SEO 页面

### 8.2 内容生产与发布系统

目标：支撑“详情、榜单、场景、专题、福利页”持续更新。

最小能力：

- 工具详情内容模板化
- 榜单支持人工策展和规则生成
- 场景页支持主推工具 + 备选工具配置
- 专题文章支持富文本/Markdown 编辑
- 支持草稿、待审、已发布状态流转
- 支持定时发布和更新记录

建议采用“结构化内容 + 富文本补充”的方式：

- 工具详情、榜单、场景采用结构化字段驱动
- 专题文章、深度评测采用富文本或 MDX 驱动
- 前台统一由渲染层拼装输出，避免后期内容结构失控

### 8.3 搜索与推荐系统

目标：把首页 AI 搜索从 Demo 交互升级为可解释、可追踪、可优化的推荐链路。

首版推荐采用三段式：

1. **规则识别**：判断是精确工具查询、场景查询、比较查询还是泛搜索
2. **候选召回**：从站内工具、标签、场景、榜单中召回候选项
3. **AI 重排与理由生成**：使用 LLM 对候选集排序，并输出简短理由

这样设计的好处：

- 不依赖大模型直接“自由发挥”
- 保证推荐结果只来自站内可控数据
- 可以记录召回与点击，后续优化更容易
- 未来可自然升级到 embedding / rerank 方案

推荐返回结构建议升级为：

- `query`
- `intent_type`
- `matched_tools`
- `matched_scenarios`
- `recommended_tools`
- `reason`
- `confidence`
- `trace_id`

### 8.4 榜单与场景引擎

目标：让榜单和场景不再只是页面，而是平台的“内容分发器”和“SEO 入口”。

首版榜单生成策略：

- 热门榜：人工策展为主
- 分类榜：评分 + 标签规则生成
- 场景榜：由场景配置生成
- 专题榜：运营手工维护

首版不追求复杂算法，核心是“可解释”和“可运营”。每个榜单都应保存：

- 榜单规则
- 入榜工具
- 排名理由
- 生效时间
- 维护人

### 8.5 商业合作与福利模块

目标：跑通“平台引流 -> 工具合作 -> 用户获利”的最小商业闭环。

最小能力：

- 为工具配置福利信息
- 支持专属链接、兑换码、预约表单三种模式
- 支持领取页和跳转页
- 支持记录来源页面、UTM、用户联系方式
- 支持统计点击、领取、提交、转化

建议把福利当成独立实体 `BenefitOffer`，而不是硬编码在工具详情页里。这样后续可以：

- 一个工具配置多个福利
- 针对不同渠道配置不同活动
- 做时效控制和 AB 测试
- 与未来的商单/活动系统衔接

### 8.6 后台管理系统

目标：让平台从“开发写死内容”切换到“运营驱动内容”。

MVO 后台最少需要以下菜单：

- 工具管理
- 榜单管理
- 场景管理
- 专题管理
- 福利管理
- 导入任务
- 审核发布
- 推荐日志
- 线索管理
- 数据概览

MVO 阶段建议只设三种角色：

- `admin`：系统配置、权限和发布
- `editor`：内容编辑、审核和上线
- `operator`：福利配置、线索处理、基础分析

## 9. 关键流程设计

### 9.1 工具入库流程

```text
Excel/手工录入
  -> 创建 ToolSnapshot
  -> 映射到 Tool 草稿
  -> AI 补全文案草稿
  -> 编辑审核
  -> 发布 Tool
  -> 自动更新详情页/搜索索引/榜单候选池
```

### 9.2 AI 推荐流程

```text
用户输入 query
  -> 意图识别
  -> 站内候选召回
  -> LLM 重排并生成推荐理由
  -> 返回工具卡片
  -> 用户点击详情/福利
  -> 记录 RecommendationLog 与点击事件
```

### 9.3 福利领取流程

```text
用户浏览工具详情页/专题页
  -> 点击福利
  -> 进入跳转页或领取页
  -> 记录来源、渠道、工具、活动
  -> 用户领取/提交表单
  -> 生成 Lead
  -> 运营跟进或跳转厂商落地页
```

## 10. 阶段性实施路线

### 阶段 0：资产整理与技术债清理（3-5 天）

目标：把 Demo 从“能演示”整理到“可迁移”。

交付：

- 梳理现有组件和页面可复用清单
- 统一中英文编码与文件格式
- 统一设计 token、路由命名、数据字段命名
- 明确迁移目录结构

### 阶段 1：MVO 数据底座与后台骨架（1-2 周）

目标：先搭出“可运营”的最低底座。

交付：

- 数据库 schema
- 工具/场景/榜单/福利核心表
- 后台登录
- 工具管理 CRUD
- Excel 导入
- 发布状态流转

验收：

- 可导入 100+ 工具草稿
- 可人工审核并发布工具详情
- 前台页面不再依赖本地 JSON

### 阶段 2：前台重构与 SEO 能力（1-2 周）

目标：让前台从 Demo 页变成可被搜索、可被运营的产品页。

交付：

- 首页、榜单、工具详情、场景页迁移到正式框架
- 动态路由和静态生成
- metadata、sitemap、robots、结构化数据
- 专题页模板
- 可发布的福利页/落地页

验收：

- 详情页、榜单页、场景页都可独立发布
- 具备基础 SEO 输出能力
- 可承接首批内容分发

## 11. ADR 关键决策

### ADR-001：采用模块化单体，不采用微服务

**状态**：Accepted

**背景**：
当前团队重点在冷启动和业务验证，不在高并发拆分。前台、后台、推荐、内容和合作链路都还会频繁联动调整。

**决策**：
MVO 阶段采用模块化单体，统一前台、后台、API 和任务能力。

**正向影响**：

- 研发速度快
- 系统复杂度低
- 上线和维护成本低

**负向影响**：

- 后期模块膨胀后需要治理边界
- 高负载任务要额外拆后台任务

### ADR-002：采用结构化数据库作为主内容源，不继续使用本地 JSON

**状态**：Accepted

**背景**：
Demo 的 JSON 适合展示，不适合审核、更新、发布、统计和溯源。

**决策**：
使用关系型数据库保存 Tool、Scenario、Ranking、Benefit、Lead 等核心实体，本地 JSON 只保留为样例数据。

**正向影响**：

- 支持运营和发布流程
- 支持搜索、推荐、埋点关联
- 支持后续后台扩展

**负向影响**：

- 需要补齐 schema 和迁移脚本
- 开发初期复杂度略高于纯静态站

### ADR-003：推荐采用“规则召回 + AI 重排”，不直接全量交给模型

**状态**：Accepted

**背景**：
平台要求推荐结果可信、可控、只输出站内已收录工具。

**决策**：
先由站内规则召回候选，再由模型做排序和理由生成。

**正向影响**：

- 可控性高
- 幻觉少
- 结果可追踪

**负向影响**：

- 首期需要维护标签和召回规则
- 召回质量会影响最终推荐质量

### ADR-004：内容生产采用“AI 起草 + 人工审核”，不做全自动发布

**状态**：Accepted

**背景**：
平台公信力来自评测质量，而不是内容数量。

**决策**：
允许 AI 生成摘要、优缺点、专题草稿，但所有对外内容必须经过人工审核发布。

**正向影响**：

- 控制内容质量
- 降低事实错误风险
- 便于建立平台调性

**负向影响**：

- 内容产出速度受编辑资源限制
- 需要后台审核流程

### ADR-005：前台必须支持 SSR/SSG 和 SEO 元数据输出

**状态**：Accepted

**背景**：
愿景明确要求依靠内容和流量驱动增长，纯 CSR Demo 不够。

**决策**：
正式 MVO 前台需采用支持 SSR/SSG、metadata、sitemap、结构化数据输出的方案。

**正向影响**：

- 更适合 SEO
- 首屏性能更可控
- 页面可独立沉淀流量

**负向影响**：

- 需要迁移现有路由和数据获取方式

## 12. 非功能要求

### 12.1 性能

- 公网页面首屏加载时间：`< 2.5s`（国内正常网络）
- 工具详情页与榜单页服务端渲染时间：`< 500ms`
- 推荐接口：规则召回 `p95 < 300ms`，AI 完整响应 `p95 < 4s`

### 12.2 可扩展性

- MVO 阶段支撑 `1000` 个工具、`5000` 篇内容规模
- 日活目标先按 `1000-5000 UV/day` 设计
- 支持未来接入独立搜索、独立推荐、更多内容类型

### 12.3 可靠性

- 目标可用性：`99.5%`
- 推荐接口失败时必须有规则兜底结果
- 内容发布必须支持回滚到上一个已发布版本

### 12.4 安全

- 前台匿名访问，后台强认证
- 所有福利领取表单要做频控和防刷
- 外部模型调用要有超时、降级和日志

### 12.5 可维护性

- 必须有错误监控、访问日志和关键事件埋点
- 所有核心实体都要有 `created_at / updated_at / operator`
- 所有 AI 输出必须标记生成来源和最后审核人

## 13. 当前代码到 MVO 的迁移建议

### 保留

- 页面信息架构
- UI 组件结构
- 设计语言
- 现有工具和场景样例

### 重做

- 路由体系
- 数据获取方式
- mock API
- 数据存储方式
- 推荐逻辑
- 内容来源和发布机制

### 优先补齐的技术债

1. 文档与代码技术栈不一致
2. 中文编码混杂，后续导入和编辑风险高
3. 数据文件仍是静态样例，不可运营
4. 推荐逻辑不可配置、不可追踪、不可复盘

### 建议的迁移方式：不是推倒，而是双轨渐进

建议不要直接在当前 Demo 上原地大修，也不要一次性重写全部页面。更稳妥的方式是：

1. 新建正式主工程，以 `Next.js App Router` 作为唯一主线
2. 现有 Demo 保留为参考实现和 UI 资产来源
3. 先迁移公共样式、基础组件、领域类型
4. 再按页面优先级迁移：首页 -> 工具详情 -> 榜单 -> 场景 -> 后台
5. 新功能只写在正式主工程，旧 Demo 不再继续加业务能力

这样做的结果是：

- 你不会失去现有 Demo 资产
- 你也不会被 Demo 架构长期绑住
- 整个系统从第一天开始就走在正式框架上

## 14. 建议的首轮交付清单

如果从今天开始立项，建议先只锁定以下首轮目标：

1. 用正式 schema 接住 100+ 工具数据
2. 先搭建 `Next.js + PostgreSQL + Prisma` 正式主工程
3. 上线后台工具管理和发布
4. 逐页重建首页、工具详情、榜单页、场景页
5. 上线规则召回 + AI 重排推荐接口
6. 上线福利页与线索收集
7. 打通埋点和基础数据看板

做到这一步，就不是“一个更完整的 Demo”，而是一个能开始真实验证增长和合作的 MVO。

## 15. 最终建议

以愿景为核心，首版不要继续定义成“内容展示 Demo”，而要定义成“AI 工具评测与转化平台的最小可运营底座”。

最关键的技术策略不是继续优化当前 Demo，而是先确认正式主框架，然后把下面三件事搭稳：

- 正式主框架：`Next.js App Router + PostgreSQL + Prisma`
- 统一领域模型
- 建立内容与发布系统
- 建立推荐与商业转化闭环

只要正式主框架和这三件事搭稳，前台 UI、模型供应商、搜索引擎、后台技术栈都可以渐进替换；如果主框架不先定，继续堆页面和交互会一直停留在 Demo 思路里，很难接近《愿景.md》的最终目标。
